# Three Files Summary: PRD → YAML → HTML
## How They Translate and Relate

---

## Overview

You now have three complementary files that represent different **levels of specification**:

1. **PRD (Markdown)** — Detailed technical specification for developers
2. **YAML** — Structured configuration/metadata representation
3. **HTML** — Visual mockup showing Phase 1 interface

Each file serves a different purpose and audience.

---

## File 1: PRD (MOOD_BOARD_PRD_PHASES_1_6.md)

### Purpose
**Complete technical specification for Claude Code or a developer to implement**

### Contains
- Phase-by-phase breakdown (Phases 1–6)
- For each phase:
  - Goal statement
  - New components to build
  - API endpoints to create
  - Hooks/utilities needed
  - Implementation details
  - Success metrics
  
- Database schema updates
- Environment variables needed
- Testing strategy
- Deployment checklist

### Audience
- **Claude Code** (AI coding agent)
- Developers implementing the feature
- Project managers tracking progress

### Example Content
```markdown
## PHASE 1: MOOD SELECTION (WEEKS 1–2)

### 1.1 New Home Screen Component
File: `src/components/MoodSelection.tsx`
Features:
- Single text input: "What are you building?"
- Displays 3 generated mood cards
- Teaching tooltip below
...
```

### How to Use It
- Give to Claude Code as the main instruction
- Reference specific phases when tasking
- Use success metrics to validate completion

---

## File 2: YAML (mood-board-config.yaml)

### Purpose
**Structured, machine-readable representation of the entire project**

### Contains
- Project metadata (name, version, links)
- Technology stack (frontend, backend, external services)
- 6 phases with:
  - Components (file paths, purposes)
  - API endpoints (method, path, request/response structure)
  - Hooks (file paths, methods)
  - Hours estimate
  - Success metrics
  
- Database schema (tables, columns)
- State management (context fields)
- Type definitions
- Environment variables
- Testing strategy
- Deployment configuration

### Audience
- **Automation tools** (could parse YAML to generate code scaffolding)
- **CI/CD pipelines** (could reference for build targets)
- **Project management tools** (could ingest for timeline tracking)
- **Documentation generators** (could auto-create API docs)

### Example Content
```yaml
phases:
  phase_1:
    name: "Mood Selection"
    duration_weeks: 2
    components:
      - name: "MoodSelection"
        file: "src/components/MoodSelection.tsx"
        purpose: "Home screen with input + mood card display"
    api_endpoints:
      - method: "POST"
        path: "/api/generate-moods"
        purpose: "Generate 3 context-aware moods from user input"
```

### How to Use It
- **Validation**: Check if implementation matches YAML specification
- **Scaffolding**: Use to auto-generate file structure
- **Tracking**: Track completed components against phase timeline
- **API contracts**: Reference for API request/response validation
- **CI/CD**: Could be parsed to set up build stages

## File 3: HTML (mood-board-phase1-mockup.html)

### Purpose
**Visual, interactive mockup of Phase 1 UI**

### Contains
- Fully styled HTML/CSS with:
  - Header (logo + tagline)
  - Step title + subtitle
  - Input textarea for project description
  - Submit button
  - 3 mood cards (with emoji, name, description, color swatches, "best for" label)
  - Selected state with checkmark
  - Teaching tooltip with design principle
  - Loading overlay
  - Empty state
  
- JavaScript for:
  - Input validation
  - Mock mood generation (with 1.5s delay)
  - Card rendering
  - Mood selection logic
  - Event handling

### Audience
- **Designers** (review visual mockup, give feedback)
- **Product managers** (see exact UI/UX flow)
- **QA testers** (use as reference for acceptance testing)
- **Developers** (reference for CSS/component structure)
- **Stakeholders** (understand what Phase 1 will look like)

### Features Demonstrated
- Responsive design (works on mobile, tablet, desktop)
- CSS variables for theming (easy to reskin)
- Smooth animations (card entries, hover states, loading spinner)
- Accessibility (semantic HTML, readable fonts, sufficient contrast)
- Error handling (input validation, friendly error messages)

### Example Usage
```
1. Open HTML file in browser
2. Type a project description (e.g., "A mobile app for meditation")
3. Click "Generate" or press Enter
4. See 3 mock mood cards appear (with 1.5s delay simulating API)
5. Click a mood card to select it

```

## Why All Three?

### PRD Alone
- ✓ Complete & detailed
- ❌ Hard to parse automatically
- ❌ Doesn't show visual representation
- ❌ Not optimized for machine-reading

### YAML Alone
- ✓ Machine-readable
- ✓ Easy to validate against
- ❌ Not detailed enough for implementation
- ❌ Doesn't show visual representation

### HTML Alone
- ✓ Shows final product
- ✓ Stakeholder-friendly
- ❌ Missing implementation details
- ❌ Missing backend specification

### All Three Together
- ✓ **PRD**: Tell Claude Code exactly what to build
- ✓ **YAML**: Track progress, validate, auto-generate structure
- ✓ **HTML**: Show stakeholders what it looks like
- ✓ **Cross-reference**: YAML components map to PRD sections map to HTML elements

## Example Workflow

### Day 1: Planning
1. Review YAML to understand scope & phases
2. Show HTML mockup to stakeholders for buy-in
3. Get approval to proceed

### Day 2-3: Phase 1 Implementation
1. Give Claude Code the PRD + Phase 1 section
2. Claude Code builds components listed in YAML
3. Reference HTML mockup as design specification
4. Validate against YAML API contract

### Day 4: Code Review
1. Check PRD success metrics:
   - "Generate 3 moods in <3 seconds" ✓
   - "Valid JSON response" ✓
2. Verify component files match YAML paths
3. Test API response structure against YAML schema

### Day 5: Handoff
1. PRD goes to next phase developer
2. YAML updated with completion status
3. HTML can be replaced with actual implementation screenshot

---

## Key Insight

**These files form a specification stack:**

- **YAML** = The skeletal blueprint (what exists, where it is)
- **PRD** = The detailed instructions (how to build it)
- **HTML** = The visual confirmation (what it looks like when done)

Working with all three gives you:
- Clarity for humans (PRD)
- Validation for machines (YAML)
- Confidence for stakeholders (HTML)
